# team-vault 소개 웹사이트 설계 문서

- **날짜**: 2026-07-04
- **상태**: 설계 승인됨 (구현 전)
- **작성 경위**: 브레인스토밍 4문답 → 3안 비교 → A안(순수 정적 원페이지) 채택 + 사용자 추가 요구(상호작용·그라데이션·확장 슬롯) 반영

---

## 1. 목적·산출물

**산출물 2종:**
1. **소개 웹사이트** — 방금 구축한 team-vault(팀 프로젝트 옵시디언 볼트 시스템)를 소개하는
   상호작용 중심 원페이지. Vercel 공개 배포.
2. **구축 기록 노트** — Brain Trinity 볼트(`<개인 지식 볼트>`)에
   "이렇게 만들었고 이렇게 쓴다" 지식 노트 저장 (웹사이트 링크 포함).

**독자(둘 다):**
- 포트폴리오 열람자(대회·생기부·외부) — "내가 설계한 팀 운영 시스템" 작품 소개
- 합류할 팀원 후보 — 시스템 이해·온보딩 입구

## 2. 채택 접근 (A안)

- **의존성 0 순수 정적**: `index.html` + `styles.css` + `main.js` + `data.js` 4파일
- 작업 환경: `<프로젝트 폴더>` + my-website-harness 복사
  (하네스 규칙 적용: main 직접 작업 금지 → 작업 브랜치, 커밋·푸시는 사용자 승인 후)
- 배포: GitHub public repo + Vercel 정적 배포
- 기각: B안 Next.js(원페이지에 과함·YAGNI), C안 Obsidian Publish(유료·Bases 미렌더·디자인 통제 불가)

## 3. 디자인 방향 (의도적 선택 — anti-template)

- **다크 퍼플 + 오로라 그라데이션**: 소개 대상이 옵시디언 기반 시스템이므로 옵시디언의
  시각 아이덴티티(딥 퍼플)를 의도적으로 계승. 기본값 다크가 아니라 제품이 원하는 방향.
- 그라데이션 사용처(사용자 요구): ①히어로 오로라 배경(보라→청록, 느린 흐름)
  ②핵심 키워드 그라데이션 텍스트 ③카드 호버 그라데이션 보더
- 금지: "가운데 제목 + 그라데이션 블롭 + 범용 CTA" 템플릿 히어로, 균일 카드 그리드
- 필수 품질(웹 규칙 4개+): 스케일 대비 위계 / 의도적 리듬 / 레이어링·깊이 / 모션이 흐름을 설명
- 모든 모션 `prefers-reduced-motion` 대응 — 끄면 정적으로도 완결된 콘텐츠

## 4. 페이지 구조 & 인터랙션 지도 (스크롤 원페이지, 7섹션)

| # | 섹션 | 내용 | 상호작용 |
|---|---|---|---|
| ① | Hero | 시스템 이름 + 한 줄 정의 | 마우스 반응 그래프 노드(canvas, 커서 근처 노드·연결선 밝아짐) + 오로라 그라데이션 |
| ② | 문제 | "팀의 기록은 카톡·머릿속·스크롤에 흩어진다" | 스크롤 진행에 따라 흩어진 메모 조각이 폴더로 정렬 |
| ③ | 구조 | 콘텐츠 폴더 9종(+templates·scripts·attachments·docs 인프라 4) + 3-시스템 경계(개인/팀/repo) | 클릭형 폴더 트리 → 옆에 설명 카드 |
| ④ | **라이브 데모 ★** | 태스크가 대시보드가 되는 과정 | 방문자가 가짜 태스크 입력(제목·담당·마감) → frontmatter 코드 타이핑 생성 → 대시보드 표에 착지. 전부 브라우저 시뮬레이션 |
| ⑤ | 온보딩 L0~L3 | "전원이 파워유저일 필요 없다" | 탭 스텝퍼 — 트랙별 역할 카드 전환 |
| ⑥ | **확장 슬롯** | 다음 시스템 티저 | `data.js`의 `projects` 배열 렌더. 현재: "🚧 제작 중 — 개인 맞춤 볼트 스타터" Coming Soon 카드 1장(사용자 확정). **Vault Starter 완성 시 배열에 객체 1개 추가로 카드 즉시 생성** |
| ⑦ | 각주 | 기술 스택·설계 원칙·만든 사람(성민혁) | 호버 디테일 |

- 인터랙션 철학: 장식이 아니라 **시스템 작동 원리(frontmatter 계약 → Bases 집계)를 체험**시키는 수단. ④가 사이트의 존재 이유.
- 사이트 내 시스템 호칭: "team-vault" (추후 회사명 확정 시 교체 가능하도록 `data.js`에 이름 상수화)

## 5. 기술 구현

- 그래프 노드: `<canvas>` (약 30개 노드, requestAnimationFrame, 탭 비활성 시 정지)
- 스크롤 연출: IntersectionObserver (스크롤 핸들러 금지)
- 애니메이션: CSS `transform`/`opacity`만 (컴포지터 친화 규칙)
- 콘텐츠 데이터: `data.js`에 배열/객체로 분리 (폴더 트리, 온보딩 트랙, 확장 슬롯 projects, 데모 예시)
- 성능 예산: **JS < 80kb, CSS < 25kb** (v1은 15kb였으나 W7 실측 20.7kb — 사용자 요구인 상호작용·그라데이션
  시각 볼륨을 반영해 상향. 실질 성능 게이트는 §8의 Lighthouse 90+ 실측)
- 폰트: 시스템 스택 + 필요 시 최대 1개 서브셋(`font-display: swap`)
- 접근성: 트리·데모·탭 전부 키보드 조작 가능(포커스 스타일 명시), 대비 WCAG AA,
  reduced-motion 시 canvas·정렬 애니메이션 정지

## 6. 공개 안전 (전체 공개 사이트)

- **미노출**: team-vault private repo URL, 팀원 실명(운영자 본인 제외), 고객·파트너 정보, 내부 경로
- 만든 사람: "성민혁" 표기 (포트폴리오 용도 — 본인 확정)
- 데모 데이터 전부 가짜(예: "제주 확장 리서치", 담당 "민혁")
- 시크릿 없음(정적 사이트 — 환경변수·API 키 자체가 없음)

## 7. 저장·배포 구조

- 폴더: `<프로젝트 폴더>` (하네스 복사 완료)
- git: master는 배포 기준선, 작업은 `feat/site-v1` 브랜치 (하네스 규칙)
- GitHub **public** repo `team-vault-website` + Vercel 연동 배포
- **Brain Trinity 기록 노트**: 구현 단계에서 볼트의 CLAUDE.md·규칙을 먼저 읽고
  그 볼트 방식(ingest 파이프라인 또는 직접 지식 노트)대로 저장. 내용: team-vault를
  왜·어떻게 만들었나 + 사용법 요약 + 웹사이트 URL. 가능하면 wiki-lint 통과.

## 8. 성공 기준

1. 320/375/768/1024/1440/1920px 전 구간 가로 스크롤 없음, 상호작용 6종(그래프·정렬·트리·데모·탭·카드 호버) 작동
2. 라이브 데모가 마우스·키보드 모두로 완주 가능 (입력→frontmatter 타이핑→대시보드 착지)
3. Lighthouse 성능·접근성 90+ (Vercel 배포 후 실측)
4. `prefers-reduced-motion` 활성 시에도 콘텐츠 완결 (애니메이션 없이 전 섹션 읽힘)
5. Vercel 공개 URL 접속 확인 + 확장 슬롯: `data.js` projects 배열에 항목 1개 추가 → 카드가 바로 생기는지 테스트
6. Brain Trinity에 기록 노트 존재 (볼트 규칙 준수)
